## Introduction

Minecraft was one of the first games that made me curious about how software actually works. I grew up playing it as a sandbox where creativity and logic overlapped. 

This project tutorial comes from that curiosity. Instead of just playing Minecraft, we are going to extend it by writing our own Java code and adding a custom item to the game. If you have ever wanted to understand how mods work under the hood or take your first step into game modding, this project is designed to make that process approachable and concrete.

This project tutorial requires a setup of the Forge MDK‚Äôs project structure and parchment. Minecraft was not built to be changed easily. The Forge MDK acts like a _translator and helper_ that lets your code talk to Minecraft safely.

Without it, Minecraft would not know:

- What your mod is
- When to load it
- How to show your item in the game

The Forge MDK gives you:

- Folders that tell you where to put your code and images
- Tools that run Minecraft with your mod installed
- Readable names for Minecraft code so it is easier to understand
- Rules so your mod does not break the game or other mods
- This can be very tedious to follow, so I have included the Forge MDK folder with the setup done. Make sure you have all the other prerequisites.

In IntelliJ IDEA, you can use autocomplete to automatically add `import` statements. This is convenient and prevents typos, but for the sake of uniformity in this curriculum, we are listing all imports explicitly at the top of the file. 

Below the first line, add the following `import` statements:


```java
package com.codedex.itemmod.item;
import com.codedex.itemmod.ItemMod;
import net.minecraft.world.item.Item;
import net.minecraftforge.eventbus.api.IEventBus;
import net.minecraftforge.registries.DeferredRegister;
import net.minecraftforge.registries.ForgeRegistries;
import net.minecraftforge.registries.RegistryObject;
```

Our first line of code will tell Minecraft that we are attempting to register a new item. This should be inside the ModItems class with correct indentation.


```java
Public static final DeferredRegister<Item> ITEMS = 
    DeferredRegister.create(ForgeRegistries.ITEMS, ItemMod.MOD_ID);
```


`DeferredRegister<Item>` is a Forge helper class that allows you to register game objects (like items) in a safe way. The `<Item>` tells Forge that we are specifically registering an item, as opposed to other things like monsters. `ItemMod.MOD_ID` tells Forge which mod namespace the item belongs to (prevents conflicts with other mods).

Next, we need to define the item‚Äôs identity in the game and ensure it is safely registered at the correct point in the mod loading process.

```java
public static final RegistryObject<Item> Codedex = ITEMS.register(name."codedex", () -> new Item(new Item.Properties()));
```

`RegistryObject<Item>` is a special container that holds a reference to the item once it is registered, acting like a ‚Äúpromise‚Äù that the item will exist after the registration process completes. The ITEMS.register method adds the item name "codedex", which **must be all lowercase to comply with Minecraft‚Äôs naming rules**. The second argument is a lambda expression, () -> new Item(new Item.Properties()), which tells Forge how to construct the item when it is time to register it, with Item.Properties() allowing you to define its characteristics, such as stack size or creative tab placement. 

For our last code within this file, we need to link our item definition to Forge‚Äôs event system, so that they appear at the correct time during Minecraft‚Äôs initialization. 

```java
public static void register(IEventBus eventBus) 
    ITEMS.register(eventBus);
}
```

The register method takes an IEventBus as a parameter, which is a pipeline that handles mod-related events. Inside the method, ITEMS.register(eventBus) tells Forge, ‚ÄúWhen it‚Äôs time to register items, run the registration code for everything in this DeferredRegister.‚Äù
