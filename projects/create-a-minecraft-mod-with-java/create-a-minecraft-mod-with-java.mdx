### But first... What is Forge MDK?

This project tutorial requires a setup of the Forge MDK‚Äôs project structure and parchment. 

The **Forge MDK** is the official starter kit you use when you want to build a Minecraft mod using the Forge mod loader. Think of the it as a pre-configured project template for modding Minecraft. üõ†Ô∏è

The name MDK, which stands for Mod Development Kit, came from the SDK, which stands for Software Developer Kit (i.e. Android SDK, iOS SDK, Unity SDK, Unreal SDK, Windows SDK, PlayStation SDK).

Minecraft was not built to be changed easily. The Forge MDK is like a translator that lets our code talk to Minecraft safely. Without Forge, Minecraft wouldn't know:

- What your mod is.
- When to load it.
- How to show your item in the game.

The Forge MDK provides:

- Folders that tell you where to put your code and images.
- Tools that run Minecraft with your mod installed.
- Readable names for Minecraft code so it is easier to understand.
- Rules so your mod does not break the game or other mods.

In IntelliJ IDEA, you can use autocomplete to automatically add `import` statements. While this is convenient and prevents typos, let's list all imports explicitly at the top of the file to maintain consistency.

Below the first line, add the following `import` statements:

```java
package com.codedex.itemmod.item;
import com.codedex.itemmod.ItemMod;
import net.minecraft.world.item.Item;
import net.minecraftforge.eventbus.api.IEventBus;
import net.minecraftforge.registries.DeferredRegister;
import net.minecraftforge.registries.ForgeRegistries;
import net.minecraftforge.registries.RegistryObject;
```

Our first line of code will tell Minecraft that we are attempting to register a new item. This should be inside the ModItems class with correct indentation.

```java
Public static final DeferredRegister<Item> ITEMS = 
  DeferredRegister.create(ForgeRegistries.ITEMS, ItemMod.MOD_ID);
```

`DeferredRegister<Item>` is a Forge helper class that allows you to register game objects (like items) in a safe way. The `<Item>` tells Forge that we are specifically registering an item, as opposed to other things like monsters. `ItemMod.MOD_ID` tells Forge which mod namespace the item belongs to (prevents conflicts with other mods).

Next, we need to define the item‚Äôs identity in the game and ensure it is safely registered at the correct point in the mod loading process.

```java
public static final RegistryObject<Item> Codedex = ITEMS.register(name."codedex", () -> new Item(new Item.Properties()));
```

`RegistryObject<Item>` is a special container that holds a reference to the item once it is registered, acting like a ‚Äúpromise‚Äù that the item will exist after the registration process completes. 

The `.register()` method adds the item name `"codedex"`, which **must be all lowercase to comply with Minecraft‚Äôs naming rules**. The second argument is a lambda expression, `() -> new Item(new Item.Properties())`, which tells Forge how to construct the item when it is time to register it, with `Item.Properties()` allowing you to define its characteristics, such as stack size or creative tab placement. 

For our last code within this file, we need to link our item definition to Forge‚Äôs event system, so that they appear at the correct time during Minecraft‚Äôs initialization. 

```java
public static void register(IEventBus eventBus) 
  ITEMS.register(eventBus);
}
```

The `.register()` method takes an IEventBus as a parameter, which is a pipeline that handles mod-related events. Inside the method, `ITEMS.register(eventBus)` tells Forge, ‚ÄúWhen it‚Äôs time to register items, run the registration code for everything in this DeferredRegister.‚Äù

Congrats! The first file is done. Your `ModItems` class should now look similar to this:

```java
package com.codedex.itemmod.item;
import com.codedex.itemmod.ItemMod;
import net.minecraft.world.item.Item;
import net.minecraftforge.eventbus.api.IEventBus;
import net.minecraftforge.registries.DeferredRegister;
import net.minecraftforge.registries.ForgetRegistries;
import net.minecraftforge.registries.RegistryObject;

public class ModItem {
  public static final DefferedRigester<item> ITEMS = 
    DefferedRegister.create(ForegeRegistres.ITEMS, ItemMod.MOD_ID);

  public static final RegistryObject<Item> Codedex = ITEMS.register(name: "codedex", () -> new Item(new Item.Properties()));

  public static void register(IEventBus eventBus) {
    ITEMS.register(eventBus);
  }
}
```

Since we are adding an item into the game, we need to make sure it‚Äôs accessible in creative. Go to the **itemmod** class within the **item** folder, and find the `.addCreative()` method. It should be empty and look like this.

```java
// Add the example block item to the building blocks tab
private void addCreative(BuildCreativeModeTabContentsEvent event)
{

}
```

Within the method, we need to ensure that when Minecraft is loading up and processing the Creative Mode code, it adds the Codedex item we had registered before. This will allow it to be accessible within the Creative Menu. Basically, we are saying ‚Äúput Codedex into the COMBAT tab.‚Äù You can also change where the item will appear, such as changing COMBAT into INGREDIENTS. 

```java
private void addCreative(BuildCreativeModeTabContentsEvent event)
{
  if (event.getTabKey() == CreativeModeTabs.COMBAT) {
    event.accept(ModItems.Codedex);
  }
}
```

This code checks if Minecraft is currently building the Ingredients creative tab, and if so, it adds your registered item (Codedex) to that tab so it appears in the creative inventory. In short: ‚ÄúIf Minecraft is currently building the combat creative tab, put Codedex into the Ingredients tab.‚Äù

We are almost done! This next section covers how we set up the item‚Äôs assets (such as how it appears in-game), make sure you follow closely.

Within **assets**, we need to create the folder **itemmod**. 

Finally, within **itemmod**, we need to create three folders:

At the end, the file structure within resources should look like this:

```
resources
  |__ assets
       |___ itemmod
              |__ lang
              |__ models
              |__ textures
```

In the **lang** folder, we will create a new file, called **en_us.json**. 

Inside it we will write:

```json

 "item.itemmod.codedex": "Codedex"
}
```


In the **models** folder, we will create another folder called **item**, and inside it, we create a file called **codedex.json**. 

Inside of that, we write:

```json
{
 "parent": "minecraft:item/generated",

   "layer0": "tutorialmod:item/codedex"
 }
}
```

### Common Mistakes to Watch Out For

When modding with Forge, beginners often run into issues like:

- Item names using uppercase letters (Minecraft requires lowercase IDs)
- Folder names not matching the mod ID exactly
- Missing registration calls in the main mod class
- Textures not appearing because of incorrect file paths

### How to Debug Your Mod

If your mod does not load or your item does not appear:
- Check the console log for red error messages
- Make sure Minecraft reaches the ‚ÄúDone‚Äù stage during loading
- Verify that your texture file name matches the JSON model exactly
- Confirm your mod ID is consistent across code and resources
